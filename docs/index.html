Manual C++ Accuracy Analysis

The manual implementation achieves 0.22% relative error, improved from the initial 28% via systematic bug fixes. The remaining gap is due to:

  • Floating-point accumulation order differences across 2073-token attention
  • BLAS summation order differs from PyTorch's ATen kernels
  • Subtle numerical divergence in multi-scale feature extraction
  • ConvTranspose2d scatter-add accumulation order

The output range closely matches PyTorch ([1.9327, 4.3891] vs [1.9355, 4.3834]), confirming the model is functionally correct. 0.22% error is well within acceptable bounds for depth estimation.

MATLAB Coder Accuracy Analysis

MATLAB Coder achieves near-bit-exact results with:

  • Max error: 5.48e-6 (single-precision rounding)
  • Mean error: 1.07e-6
  • Output range matches PyTorch exactly
  • All ops auto-generated from .pt2 graph
The ~1e-6 error is at the single-precision floating-point limit, confirming mathematically equivalent computation.

Debugging Journey: 28% → 1% → 0.22% Relative RMSE

The initial manual C++ implementation had 28% relative RMSE. Three rounds of systematic debugging reduced it to 0.22%, and optimization reduced runtime from 13.5s to 1.0s.

Step 1: Extract PyTorch Checkpoints
Generated binary checkpoints at 12 intermediate stages by manually tracing through the model: patch embedding, positional embedding, each transformer block output, and decoder feature maps.
Step 2: Build C++ Debug Harness
Created debug_main.cpp that replicates the forward pass step-by-step, comparing each intermediate buffer against PyTorch's binary checkpoint. Computes max error, RMSE, and mean for each stage.
Step 3: Identify Error Source
Initial checkpoint comparison revealed: Patch embedding (0 error), tokens (0 error), positional embedding (3.45e-3 max error) — all downstream errors traced back to this initial divergence.
Step 4: Fix Bicubic Kernel (Bug #1)
The bicubic interpolation used Catmull-Rom kernel (a=-0.5) instead of PyTorch's default (a=-0.75). Fixing this single coefficient reduced positional embedding error from 3.45e-3 to 2.34e-7 — a 15,000x improvement. All encoder errors dropped to single-precision limits.
Step 5: Fix RefineNet Flow (Bug #2)
Despite encoder accuracy, the final output still had 28% error. Graph tracing revealed the DPT decoder's RefineNet stages had wrong operation ordering: my code did merge-then-RCU1-then-RCU2, but the actual graph does RCU1(feature) → merge → RCU2. Also, out_conv applies after upsampling, not before.
Step 6: Reduce to 1.05% RMSE
After both fixes: relative RMSE dropped from 27.9% to 1.05%. Output range [1.90, 4.39] now closely matches PyTorch [1.94, 4.38]. Remaining gap traced to floating-point accumulation order.
Step 7: Fix LayerNorm eps (Bug #3)
Inspected the .pt2 graph using torch.fx and found all 28 LayerNorm nodes use eps=1e-6 (DINOv2's default), not the standard 1e-5. One-character fix in C++ reduced RMSE from 1.05% to 0.22%.
Step 8: Speed Optimization (13.5s → 1.0s)
Replaced naive 6-loop conv2d with im2col + Accelerate BLAS GEMM (13x speedup). Added pre-allocated persistent buffers (no per-pass heap churn) and cached bicubic positional embedding interpolation. Final: 1,043 ms average.

Bug #3: LayerNorm eps=1e-5 vs 1e-6

// Graph inspection (torch.fx): // torch.ops.aten.layer_norm.default(..., 1e-06) // All 28 LayerNorm nodes: eps = 1e-6 (DINOv2) // BEFORE (wrong - standard eps) float eps = 1e-5f; // AFTER (correct - DINOv2 uses 1e-6) float eps = 1e-6f;
Impact: Relative RMSE dropped from 1.05e-2 to 2.24e-3 — a 4.7x improvement. One character change. Discovered by printing the .pt2 graph node arguments with torch.fx.

Speed: im2col + BLAS Conv2d

// BEFORE: naive 6-loop conv2d: ~12,000 ms for (oc) for (oh) for (ow) for (ic) for (kh) for (kw) out[oc][oh][ow] += w[oc][ic][kh][kw] * in[ic][...]; // AFTER: im2col + BLAS GEMM: ~370 ms for all convs im2col(input, col_buf, ic, ih, iw, kh, kw, stride, pad); // col_buf: [ic*kh*kw, oh*ow] — cache friendly cblas_sgemm(CblasRowMajor, CblasNoTrans, CblasNoTrans, out_c, oh*ow, ic*kh*kw, 1.0f, weight, col_buf, 0.0f, output);
Impact: All 31 Conv2d ops 13x faster. Total inference: 13,466 ms → 1,043 ms.

Bug #1: Bicubic Kernel Coefficient

// BEFORE (wrong - Catmull-Rom, a=-0.5) if (ax <= 1.0f) return (1.5f*ax - 2.5f)*ax*ax + 1.0f; if (ax < 2.0f) return (-0.5f*ax + 2.5f)*ax*ax - 4.0f*ax + 2.0f; // AFTER (correct - PyTorch default, a=-0.75) if (ax <= 1.0f) return ((1.25f*ax - 2.25f)*ax)*ax + 1.0f; if (ax < 2.0f) return ((-0.75f*ax + 3.75f)*ax - 6.0f)*ax + 3.0f;
Impact: Positional embedding error dropped from max=3.45e-3 to max=2.34e-7 (15,000x). This single fix was the root cause of most encoder error accumulation.

Bug #2: RefineNet Operation Order

// BEFORE (wrong order) merged = upsample(prev_stage) + feature; rcu1(merged); // RCU1 on merged rcu2(rcu1_out); // RCU2 on RCU1 output // AFTER (correct - from graph trace) rcu1_out = rcu1(feature); // RCU1 on raw feature merged = out_conv(prev_stage) + rcu1_out; rcu2(merged); // RCU2 on merged // out_conv applied AFTER upsample
Impact: Output range shifted from [1.50, 3.39] to [1.90, 4.39] (matching reference [1.94, 4.38]). Without this fix, the decoder completely misrouted feature information.

Why MATLAB Coder Achieves ~4,020x Better Accuracy

MATLAB Coder achieves relative RMSE of 5.57e-7 vs Manual C++'s 2.24e-3. After all bug fixes (bicubic kernel, RefineNet order, LayerNorm eps), the gap is fundamental to the approach — not fixable bugs.

1. Exact Graph Reproduction

MATLAB Coder's loadPyTorchExportedProgram reads the .pt2 graph and generates C++ that reproduces the exact sequence of operations node-by-node — same operation order, same intermediate tensor layouts, same reduction patterns.

Manual C++ implements the same mathematical function but with different computation ordering (e.g., BLAS accumulates partial sums in a different order).

2. Float Non-Associativity

IEEE 754 float32 addition is not associative: (a+b)+c ≠ a+(b+c). In attention alone, each token sums 2073 softmax-weighted values across 6 heads.

Over 12 blocks, ~1e-7 per-op rounding differences compound to ~1e-5 at the encoder. Layer normalization then amplifies these through variance division, expanding 1e-5 encoder errors to 1e-2 decoder errors.

3. Error Amplification Chain

The error amplification path:
Bicubic interp (1e-7) →
12 ViT blocks (1e-5) →
Layer norm (1e-4, amplified by variance) →
31 decoder convs (1e-2, accumulated)

MATLAB Coder avoids this entirely by matching the exact computation graph.

Verdict

The trade-off is clear: Manual C++ is 12.3x faster (1.0s vs 13s) with 0.22% relative error; MATLAB Coder is near-exact (5.57e-7 RMSE) with automated generation in minutes.

Choose Manual C++ when latency matters: real-time depth estimation, edge deployment, or when 0.22% error is acceptable (it is for most depth applications).

Choose MATLAB Coder when accuracy must be guaranteed: safety-critical systems, bit-exact validation, or when prototyping quickly without C++ expertise. MATLAB Coder also supports OpenMP for further parallelism.

Critical Insight: BLAS Dependency Determines Portability

The Manual C++ implementation's 12.3x speed advantage over MATLAB Coder exists only because Apple Accelerate (BLAS) is available. On real embedded targets — the same ECUs that MATLAB Coder is designed for — BLAS does not exist. This fundamentally changes the deployment trade-off.

What It Would Take to Port Manual C++ to a Safety-Certified ECU

Required Changes

1. Replace BLAS
Rewrite all cblas_sgemm calls as hand-written NEON intrinsics or link ARM Compute Library (which itself requires licensing and qualification). Without BLAS, encoder runtime returns to ~13s.
2. Staticize all buffers
Replace every std::vector (34 instance) with static float buf[COMPILE_TIME_SIZE]. All sizes are known at compile time — this is feasible but tedious.
3. Embed weights as flash arrays
Convert 234 binary weight files into const float weights[] __attribute__((section(".rodata"))). Adds ~94 MB to flash image. Remove all std::ifstream code.
4. MISRA-C++ sweep
Remove lambdas, auto, thread_local, range-for. Weeks of work with a MISRA checker, plus documentation of any justified deviations.
5. Code traceability
Manually document which code lines implement which model graph nodes — required for ISO 26262 ASIL-B/C/D certification. Not feasible to do retroactively at scale.

Accuracy Floor Insight: Why 0.22% Is Irreducible

An experiment was run replacing all 31 decoder convolutions with exact-order 6-loop double-precision code to try to match PyTorch's accumulation order. Result: zero accuracy improvement, 15x slower.

The error is 100% from encoder precision: 12 transformer blocks with 2073-token attention accumulate ~1.28e-4 max error, which the decoder amplifies ~750x to the final 1.22e-2 max output error. The decoder convolution order is irrelevant. To go below 0.22% RMSE with manual C++ would require matching PyTorch's exact ATen kernel accumulation order in the encoder — effectively rebuilding MATLAB Coder from scratch.
